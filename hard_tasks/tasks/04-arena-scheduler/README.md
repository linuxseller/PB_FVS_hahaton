# 04. Arena Scheduler


## Суть

Нужно написать агента принятия решений. Judge будет отправлять текущее состояние арены, а ваш сервис должен выбрать действие.

Это задача на стратегию, оценку риска и планирование.

## Контракт решения

Контейнер слушает порт `8080`.

### Healthcheck

```http
GET /health
```

### Решение хода

```http
POST /decide
```

Пример состояния:

```json
{
  "tick": 1,
  "resources": {
    "energy": 10,
    "workers": 2,
    "budget": 100
  },
  "active_tasks": [],
  "available_tasks": [
    {
      "id": "t1",
      "reward": 50,
      "duration": 2,
      "deadline_tick": 5,
      "energy_cost": 3,
      "risk": 0.1
    }
  ]
}
```

Ответ:

```json
{
  "action": "accept",
  "task_id": "t1"
}
```

Разрешённые действия:

```text
accept
skip
```

## Что проверяется

Публичные тесты проверяют, что сервис:

- возвращает валидное действие;
- не выбирает несуществующие задачи;
- не выбирает задачу дороже доступной энергии;
- умеет выбирать более выгодные задачи.

В скрытых тестах будет полноценная симуляция с большим количеством ходов.
